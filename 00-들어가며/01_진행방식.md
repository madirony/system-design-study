# 1. 스터디 진행 방식 최종 확정

## 섹션 목적

이 섹션에서는 스터디의 전체 진행 방식을 다시 한 번 확정하고, 다음 주부터 각자 무엇을 준비해오면 되는지 안내한다.

핵심은 단순 독서형 스터디가 아니라, 먼저 직접 설계해보고 서로 질문하면서 보완하는 실습형 스터디라는 점을 전달하는 것이다.

---


이전에 공유드린 것처럼 이번 스터디는 단순히 책을 읽고 요약하는 방식보다는, 먼저 직접 설계해보고 책 내용을 보면서 보완하는 방식으로 진행하려고 합니다.

그래서 매 챕터는 기본적으로 다음 흐름으로 진행하겠습니다.

```text₩
1차 설계 → 모의면접/리뷰 → 책 학습 → 재설계
```

책 내용을 먼저 전부 읽고 정답을 맞히는 방식이 아니라, 챕터의 문제와 요구사항을 보고 각자 먼저 설계해보는 방식입니다.

완성도 높은 정답을 가져오는 것이 목적은 아니고, “왜 이런 구조를 선택했는지”를 설명하고, 서로 질문하면서 설계 감각을 키우는 것이 목적입니다.

---

## 전체 진행 방식

진행은 A/B/C팀이 한 Zoom에서 같이 발표하는 방식이 아니라, 각 팀이 별도 Zoom에서 따로 진행하는 방식으로 가져가겠습니다.

각 팀은 같은 챕터를 기준으로 진행하되, 팀 안에서 개인별 설계안을 공유하고 서로 리뷰하는 형태입니다.

즉, 공통 진행 흐름은 동일하지만 실제 토론과 리뷰는 각 팀별 Zoom에서 진행합니다.

---

## 1주차: 책을 다 읽기 전에 먼저 설계

1주차에는 책의 해설 내용을 먼저 다 읽기보다는, 해당 챕터의 문제 상황과 요구사항만 보고 개인별로 1차 설계를 준비해오면 됩니다.

각자 문제와 요구사항을 보고 “나는 이 문제를 이렇게 이해했고, 이런 구조가 필요하다고 생각했다”를 정리해오면 됩니다.

그림이 꼭 예쁘지 않아도 되고, 텍스트로 정리해도 괜찮습니다. 중요한 것은 설계의 완성도보다 “왜 그렇게 생각했는지”입니다.

### 1차 설계에 포함하면 좋은 내용

1. 요구사항 정리
2. 핵심 API 또는 기능 흐름
3. 데이터 저장 구조
4. 전체 아키텍처
5. 병목이나 장애가 날 수 있는 지점
6. 내가 이 구조를 선택한 이유
7. 고민한 트레이드오프

---

## 정리 형식

설계 정리 형식은 자유롭게 가져가겠습니다.

Notion, 문서, Excalidraw, Mermaid, Markdown 등 본인이 편한 방식으로 정리하면 됩니다.

다만 최종 정리본은 개인별로 GitHub에 올리는 방식으로 통일하겠습니다.

여기서 중요한 기준은 형식보다 내용입니다.

문서가 완벽하거나 다이어그램이 예쁘지 않아도 괜찮습니다. 대신 아래 내용은 드러나면 좋겠습니다.

1. 요구사항을 어떻게 해석했는지
2. 어떤 구조로 설계했는지
3. 왜 그 구조를 선택했는지
4. 어떤 병목이나 장애 가능성을 고려했는지
5. 어떤 트레이드오프가 있었는지

---

## 스터디 당일: 모의면접/리뷰

스터디 당일에는 각 팀 Zoom에서 개인별 설계안을 공유하고, 서로 면접관처럼 질문하는 방식으로 리뷰합니다.

발표자는 본인의 설계 의도를 설명하고, 듣는 사람은 질문을 통해 설계의 빈틈이나 개선 포인트를 같이 찾아보면 됩니다.

### 리뷰 때 사용할 수 있는 질문 예시

- 트래픽이 늘어나면 어디가 먼저 병목이 될까요?
- Redis나 DB 장애가 나면 어떻게 처리할까요?
- 이 구조에서 단일 장애 지점은 어디인가요?
- 데이터 정합성은 어디까지 보장하나요?
- 캐시와 DB 데이터가 다르면 무엇을 기준으로 하나요?
- DB를 이렇게 나눈 이유가 있나요?
- 왜 이 저장소를 선택했나요?
- 동시 요청이 몰리면 어떤 문제가 생길 수 있나요?
- 장애가 발생했을 때 재처리는 어떻게 할 수 있나요?

### 팀별 리뷰 방식

```text
발표하는 사람: 설계 의도 설명
듣는 사람: 질문 1개 이상 준비
팀 전체: 더 나은 구조가 있는지 함께 토론
```

---

## 2주차: 책 학습 후 재설계

2주차에는 책의 나머지 내용을 읽고, 본인이 처음 설계했던 내용과 비교합니다.

책에서 제안한 구조와 내 설계가 어떻게 달랐는지, 내가 놓친 요구사항이나 병목 지점은 무엇이었는지 확인한 뒤 보완 설계를 진행합니다.

보완된 설계도 개인별로 GitHub에 올리면 됩니다.

### 2주차에 정리하면 좋은 내용

1. 처음 설계에서 부족했던 부분
2. 책을 읽고 새로 알게 된 부분
3. 책의 설계와 내 설계가 달랐던 지점
4. 보완한 최종 설계
5. 다음 설계 때 가져갈 교훈

---

## 오늘 확정할 내용

이 섹션에서는 아래 내용을 확정하면 됩니다.

1. A/B/C 팀별 Zoom으로 따로 진행한다.
2. 1차 설계는 개인별로 준비한다.
3. 설계 정리 형식은 자유롭게 가져간다.
4. 개인별 정리본은 GitHub에 업로드한다.
5. 2주차에는 책 학습 후 보완 설계를 다시 업로드한다.

---


정리하면, 이번 스터디는 책 내용을 그대로 요약하는 방식이 아니라 직접 설계해보고, 팀 안에서 서로 질문하면서 보완하는 방식으로 진행하겠습니다.

1주차에는 책 해설을 먼저 다 보기보다 문제와 요구사항을 보고 개인별로 1차 설계를 준비해옵니다.

스터디 시간에는 A/B/C 각 팀별 Zoom에서 설계안을 공유하고, 서로 면접관처럼 질문하면서 리뷰합니다.

2주차에는 책 내용을 읽고 처음 설계와 비교한 뒤, 부족했던 부분을 보완해서 재설계합니다.

정리 형식은 자유롭게 가져가되, 개인별 최종 정리본은 GitHub에 올리는 방식으로 통일하겠습니다.

이 스터디의 목적은 완벽한 정답을 만드는 것이 아니라, “왜 이렇게 설계했는지”를 설명하고 질문을 통해 설계를 개선하는 감각을 키우는 것입니다.
