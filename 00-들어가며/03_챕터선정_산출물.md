# 4. 첫 챕터 선정 ~ 5. 주차별 산출물 정하기

## 섹션 목적

이 섹션에서는 실제 실습형 스터디의 첫 챕터와 주차별 준비물을 확정합니다.

핵심은 네 가지입니다.

1. 첫 챕터는 **6장 키-값 저장소 설계**로 진행한다.
2. 1주차에는 책 해설을 먼저 보기보다 각자 1차 설계를 준비한다.
3. 2주차에는 책 내용을 읽고 기존 설계를 보완해 재설계한다.
4. 산출물은 개인별로 GitHub에 업로드한다.

---

# 4. 첫 챕터 선정


실제 실습형 스터디는 6장부터 시작하기로 했기 때문에,  
첫 번째 챕터는 **6장 키-값 저장소 설계**로 진행합니다.

처리율 제한 장치는 처음 실습형 설계를 시작하기에 적절한 주제입니다.

이유는 다음과 같습니다.

1. 요구사항을 잡기 비교적 명확하다.
2. API, Redis, 분산 환경, 동시성, 장애 상황 같은 백엔드 설계 포인트를 함께 다루기 좋다.
3. 설계 난이도가 너무 높지 않으면서도 토론할 요소가 충분하다.
4. 면접형 시스템 설계 연습에 적합하다.

따라서 첫 2주 동안은 처리율 제한 장치를 주제로 진행합니다.

- 1주차: 각자 1차 설계 준비 및 팀별 리뷰
- 2주차: 책 내용 학습 후 보완 설계

이후 챕터는 기본적으로 책 순서대로 진행하되,  
필요하면 난이도나 일정에 따라 중간에 조정할 수 있습니다.

다음으로 첫 챕터와 주차별 산출물에 대해 안내드리겠습니다.

실제 실습형 스터디는 6장부터 시작하기로 했기 때문에,  
첫 번째 챕터는 **6장 키-값 저장소 설계**로 진행하겠습니다.

이 주제는 처음 실습형 설계를 시작하기에 적절하다고 생각합니다.

요구사항을 잡기 비교적 명확하고,  
API, Redis, 분산 환경, 동시성, 장애 상황 같은 백엔드 설계 포인트를 같이 이야기하기 좋기 때문입니다.

그래서 첫 2주 동안은 처리율 제한 장치를 주제로 진행하겠습니다.

1주차에는 각자 1차 설계를 준비해오고,  
2주차에는 책 내용을 바탕으로 기존 설계를 보완하는 방식으로 진행하겠습니다.

이후 챕터들은 기본적으로 책 순서대로 진행하되,  
필요하면 난이도나 일정에 따라 중간에 조정하겠습니다.

# 5. 주차별 산출물 정하기

주차별 산출물을 정해두는 이유는 스터디가 단순한 대화로 끝나지 않고,  
각자의 설계 사고 과정이 기록으로 남도록 하기 위함입니다.

중요한 것은 문서를 예쁘게 만드는 것이 아니라,  
**왜 그렇게 설계했는지 설명할 수 있는 상태로 준비하는 것**입니다.

---

## 1주차 산출물: 1차 설계안

1주차에는 책의 전체 해설을 먼저 보기보다는,  
문제와 요구사항을 보고 각자 1차 설계안을 준비합니다.

각자 정리해올 내용은 아래 정도면 됩니다.

1. 요구사항 정리
2. 핵심 API 또는 기능 흐름
3. 데이터 저장 구조
4. 전체 아키텍처
5. 병목 또는 장애 가능 지점
6. 이 구조를 선택한 이유

정리 형식은 자유입니다.

- Notion
- Markdown
- 문서
- Excalidraw
- Mermaid
- 기타 본인이 편한 방식

그림이 꼭 예쁘지 않아도 되고, 텍스트로 정리해도 괜찮습니다.

중요한 것은 아래 질문에 답할 수 있는 상태입니다.

- 나는 이 문제를 어떻게 이해했는가?
- 어떤 요구사항을 중요하게 봤는가?
- 왜 이 구조를 선택했는가?
- 이 구조는 어디에서 병목이 생길 수 있는가?
- 장애가 발생하면 어떤 부분이 영향을 받는가?

---

## 2주차 산출물: 책 기반 보완 설계안

2주차에는 책의 나머지 내용을 읽고,  
본인이 1주차에 설계했던 내용과 비교합니다.

그리고 부족했던 부분을 보완해서 재설계합니다.

2주차 정리 내용은 아래 정도면 됩니다.

1. 기존 1차 설계에서 부족했던 점
2. 책을 읽고 새로 알게 된 핵심 포인트
3. 보완한 최종 아키텍처
4. 다음 설계 때 가져갈 교훈

2주차에는 책 내용을 정답처럼 그대로 옮기는 것보다,  
내 설계와 책의 설계를 비교하는 것이 중요합니다.

비교할 때는 아래 질문을 기준으로 보면 좋습니다.

- 내 설계에서 놓친 것은 무엇인가?
- 책의 설계가 더 나은 이유는 무엇인가?
- 책의 설계도 실제 운영 환경에서는 어떤 한계가 있을까?
- 우리 서비스라면 다르게 설계할 부분은 무엇인가?

---

## GitHub 업로드 기준

1주차 1차 설계안과 2주차 보완 설계안은 개인별로 GitHub에 업로드합니다.

정리 형식은 자유롭게 가져가되,  
GitHub에는 개인별 산출물이 확인될 수 있도록 업로드합니다.

파일명이나 디렉토리 구조는 운영 편의를 위해 간단히 맞추겠습니다.

예시는 아래와 같습니다.

```text
chapter04-rate-limiter/
  ├── week1-initial-design.md
  └── week2-redesign.md
```

또는 첫 스터디에서는 더 간단하게 README 하나에 누적 작성할 수도 있습니다.

```text
chapter04-rate-limiter/
  └── README.md
```

첫 챕터는 부담을 줄이기 위해  
README.md 하나에 1차 설계와 보완 설계를 누적 작성하는 방식도 괜찮습니다.

# 스터디에서 함께 키우면 좋은 설계 역량

이번 스터디는 단순히 책 내용을 아는 것보다,  
**설계를 설명하는 힘**을 키우는 방향으로 진행하면 좋겠습니다.

책에 나온 아키텍처를 외우는 것이 아니라,  
문제를 보고 요구사항을 추상화하고, 병목을 예상하고, 확장 가능한 구조로 발전시키는 설계 사고를 함께 연습하는 것이 목표입니다.

---

## 1. 요구사항을 추상화하는 힘

시스템 설계에서 중요한 것은 바로 구조를 그리는 것이 아니라,  
문제를 어떤 단위로 쪼개서 볼 것인지 정하는 것입니다.

예를 들어 처리율 제한 장치 설계라면  
바로 Redis부터 떠올리기보다 먼저 아래 질문을 던져볼 수 있습니다.

```text
누가 제한 대상인가?
- 사용자 기준인가?
- IP 기준인가?
- API Key 기준인가?
- 서비스 또는 테넌트 기준인가?

무엇을 제한할 것인가?
- 초당 요청 수
- 분당 요청 수
- 특정 API 호출 수
- 쓰기 요청만 제한

제한을 넘으면 어떻게 할 것인가?
- 429 응답
- 큐잉
- 지연 처리
- 차단
```

이처럼 구현 기술보다 정책과 기준을 먼저 뽑아내는 연습이 필요합니다.

---

## 2. 설계의 확장성을 넓히는 힘

처음부터 거대한 구조를 만드는 것이 확장성은 아닙니다.

확장성은 요구사항이 커졌을 때,  
어디를 바꾸면 되는지 명확한 구조에 가깝습니다.

스터디에서는 매번 아래 질문을 던져보면 좋습니다.

```text
- 사용자가 10배 늘어나면 어디가 먼저 병목이 될까?
- 단일 서버에서 분산 환경으로 바뀌면 무엇이 깨질까?
- DB 하나로 버티지 못하면 어떤 기준으로 나눌까?
- 캐시를 붙이면 정합성 문제는 어디서 생길까?
- 장애가 났을 때 전체 시스템이 멈추는가, 일부만 실패하는가?
```

이 질문을 반복하면 설계를 더 넓게 보는 감각을 키울 수 있습니다.

---

## 3. 트레이드오프를 말하는 힘

좋은 설계는 “이게 정답입니다”라고 말하는 것이 아니라,  
왜 이 선택을 했고 무엇을 포기했는지 설명할 수 있어야 합니다.

예를 들어 Redis를 사용하는 경우에도 아래처럼 설명할 수 있어야 합니다.

```text
Redis를 사용했다.
→ 빠른 카운팅과 TTL 처리가 가능하다.
→ 대신 Redis 장애 시 제한 정책이 흔들릴 수 있다.
→ 그래서 fail-open / fail-close 정책을 정해야 한다.
```

이런 식으로 선택의 장점과 한계를 함께 설명하는 연습을 하면  
면접형 설계 답변도 훨씬 단단해집니다.

---

## 4. 책과 내 설계를 비교하는 힘

2주차에는 책 내용을 정답으로만 보지 않는 것이 좋습니다.

책에서 제안한 구조를 보면서  
내 설계와 어떤 점이 달랐는지, 어떤 요구사항을 놓쳤는지, 실제 운영 환경에서는 어떤 한계가 있을지 비교해보는 것이 중요합니다.

이렇게 해야 단순 독서가 아니라 실제 설계 훈련이 됩니다.

---

다음으로 첫 챕터와 주차별 산출물에 대해 안내드리겠습니다.

실제 실습형 스터디는 6장부터 시작하기로 했기 때문에,  
첫 번째 챕터는 **6장 키-값 저장소 설계**로 진행하겠습니다.

처리율 제한 장치는 처음 실습형 설계를 시작하기에 적절한 주제라고 생각합니다.

요구사항을 잡기 비교적 명확하고,  
API, Redis, 분산 환경, 동시성, 장애 상황 같은 백엔드 설계 포인트를 같이 이야기하기 좋기 때문입니다.

첫 2주 동안은 처리율 제한 장치를 주제로 진행하겠습니다.

1주차에는 책의 전체 해설을 먼저 보기보다는,  
문제와 요구사항을 보고 각자 1차 설계안을 준비해오면 됩니다.

준비할 내용은 요구사항 정리, 핵심 API나 기능 흐름, 데이터 저장 구조,  
전체 아키텍처, 병목이나 장애 가능 지점, 그리고 이 구조를 선택한 이유 정도입니다.

정리 형식은 자유입니다.

Notion, Markdown, 문서, Excalidraw, Mermaid 등 편한 방식으로 작성하시면 됩니다.  
그림이 꼭 예쁘지 않아도 되고, 텍스트로 정리해도 괜찮습니다.

중요한 건 문서를 예쁘게 만드는 것이 아니라,  
“왜 이렇게 설계했는지”를 설명할 수 있는 상태로 준비해오는 것입니다.

2주차에는 책의 나머지 내용을 읽고,  
본인이 1주차에 설계했던 내용과 비교합니다.

책에서 제안한 구조와 내 설계가 어떻게 달랐는지,  
내가 놓친 요구사항이나 병목 지점은 무엇이었는지 확인한 뒤  
보완 설계를 진행합니다.

2주차에는 기존 설계에서 부족했던 점,  
책에서 배운 핵심 포인트,  
보완한 최종 아키텍처,  
다음 설계 때 가져갈 교훈 정도를 정리하면 됩니다.

1주차 1차 설계안과 2주차 보완 설계안은  
개인별로 GitHub에 업로드하는 방식으로 진행하겠습니다.

추가로, 이번 스터디에서는 책 내용을 외우는 것보다  
설계 사고를 키우는 데 초점을 두면 좋겠습니다.

특히 요구사항을 추상화하는 방법,  
병목을 예상하는 방법,  
확장 가능한 구조로 설계를 발전시키는 방법을 같이 연습해보면 좋을 것 같습니다.

처음부터 완벽한 아키텍처를 만드는 게 목적은 아닙니다.

“왜 이 구조를 선택했는지”,  
“이 구조는 어디까지 버틸 수 있는지”,  
“무엇을 포기한 선택인지”를 설명하는 연습을 해보려고 합니다.

그래서 각자 설계안을 볼 때도 단순히 맞다, 틀리다로 보기보다는  
요구사항 해석, 확장성, 장애 대응, 트레이드오프 관점에서 질문해주시면 좋겠습니다.
